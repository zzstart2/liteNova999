## 3. 测试用例设计

### 3.1 场景测试 — 故障转移序列 (ST)

| 用例编号 | 测试场景 | 预期结果 | 优先级 |
|---------|---------|---------|--------|
| ST-001 | 主→备故障转移（A 宕机，B 接管） | 请求不中断，B 返回成功 | P0 |
| ST-002 | 级联故障（A→B→C 依次失败，D 兜底） | 最终由 D 成功，前 3 个记录失败 | P0 |
| ST-003 | 闪断恢复（A 短暂失败 2 次后恢复） | 不触发熔断，A 继续服务 | P0 |
| ST-004 | 渐进劣化（A 错误率缓慢上升） | 健康检查标记 degraded → 降级排序 | P0 |
| ST-005 | 全面故障（所有服务商同时宕机） | AllProvidersUnavailableError | P0 |
| ST-006 | 全面故障后逐个恢复 | 最先恢复的服务商接管流量 | P0 |
| ST-007 | 多轮交替故障（A/B 交替宕机） | 始终有可用路由，无中断 | P1 |

### 3.2 集成测试 — HealthChecker + FallbackRouter 联动 (HF)

| 用例编号 | 测试场景 | 预期结果 | 优先级 |
|---------|---------|---------|--------|
| HF-001 | 健康检查结果驱动路由排序 | healthy 优先于 degraded/unhealthy | P0 |
| HF-002 | 健康状态变化触发路由切换 | A 从 healthy→unhealthy, 流量转到 B | P0 |
| HF-003 | 健康恢复后路由回切 | A 恢复 healthy, 流量回到 A | P0 |
| HF-004 | 响应时间影响排序 | 同为 healthy 时快者优先 | P1 |
| HF-005 | 健康检查 + 熔断器双重保护 | 两套机制协同工作无冲突 | P0 |

### 3.3 恢复机制测试 (RC)

| 用例编号 | 测试场景 | 预期结果 | 优先级 |
|---------|---------|---------|--------|
| RC-001 | 熔断→等待→半开探测成功→恢复 | 完整恢复周期 | P0 |
| RC-002 | 熔断→半开探测失败→重新熔断→再恢复 | 二次恢复成功 | P0 |
| RC-003 | 多服务商交错恢复 | 各自独立恢复，无干扰 | P0 |
| RC-004 | 恢复后的流量分配 | 恢复的服务商按原优先级重新参与 | P1 |
| RC-005 | 长时间熔断后恢复 | 超长 recovery_timeout 正确处理 | P1 |

### 3.4 性能测试 (PT)

| 用例编号 | 测试场景 | 预期结果 | 优先级 |
|---------|---------|---------|--------|
| PT-001 | Fallback 延迟开销（无故障基线） | 路由开销 < 1ms | P0 |
| PT-002 | 单次 Fallback 延迟（1 次失败） | 额外开销 < 2ms | P0 |
| PT-003 | 并发 200 请求 Fallback | 全部正确路由 | P0 |
| PT-004 | 5 服务商 × 1000 次请求 | 无内存泄漏，路由始终正确 | P1 |

### 3.5 边界/异常测试 (ET)

| 用例编号 | 测试场景 | 预期结果 | 优先级 |
|---------|---------|---------|--------|
| ET-001 | 所有服务商同时熔断→同时到达恢复时间 | 不死锁，探测有序执行 | P0 |
| ET-002 | 故障转移过程中新增服务商 | 不影响当前请求 | P1 |
| ET-003 | action 抛出非标准异常 (KeyboardInterrupt) | 不被吞掉 | P1 |
| ET-004 | 极端配置: 1 服务商 threshold=1 timeout=0 | 正确执行 | P1 |
| ET-005 | 服务商返回成功但极慢 | 不触发熔断，但健康检查标记 degraded | P1 |

## 4. 测试环境

- Python 3.x + pytest + pytest-asyncio
- FakeClock 控制时间流逝
- Mock action 模拟服务商行为
- 不依赖真实外部服务

## 5. 通过标准

- 所有 P0 用例通过
- P1 用例通过率 ≥ 90%
- 路由开销 P95 < 2ms
- 无死锁、内存泄漏或未处理异常
